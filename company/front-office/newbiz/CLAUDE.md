# 新規事業開発部門

## 役割
新しいプロダクト・事業のアイデア創出→検証→MVP構築→ピッチまでを担当する。
「作る前に検証する」を徹底し、無駄な開発を防ぐ。

## プロセス（リーンスタートアップ）
1. **アイデア** — 課題発見・仮説立案 → `ideas/`
2. **検証** — 市場調査・競合分析・ユーザーヒアリング → `validation/`
3. **MVP** — 最小限のプロトタイプ設計・構築 → `mvp/`
4. **ピッチ** — 事業計画・収益モデル・プレゼン資料 → `pitches/`

## ルール
- アイデアは思いついたら即 `ideas/` に記録。粗くてOK
- 検証なしに開発着手しない。必ず `validation/` で仮説検証を行う
- 検証項目: ①課題は実在するか ②お金を払う人がいるか ③自分たちに優位性はあるか
- MVPは最短2週間で出せる範囲に絞る
- 技術選定は dev部門と連携。実装が必要なら dev に依頼
- 検証結果がNGなら潔く撤退。記録は残す（学びとして）
- 完了・進捗は秘書のTODOに報告を追記

## アイデアテンプレート
```markdown
# [アイデア名]

## 課題
誰の、どんな課題を解決するか

## 仮説
こうすれば解決できるのではないか

## ターゲット
誰に売るか（具体的なペルソナ）

## 収益モデル
どうやって稼ぐか

## 競合
既存の解決策は何か、なぜそれでは不十分か

## 検証方法
どうやってこの仮説を最速で検証するか

## ステータス
idea / validating / building / launched / killed
```

## 検証テンプレート
```markdown
# [アイデア名] 検証レポート

## 仮説
## 検証方法（何をやったか）
## 結果（数字・ファクトベース）
## 判定（GO / PIVOT / KILL）
## 次のアクション
```

## フォルダ構成
- `ideas/` — アイデアメモ（1アイデア1ファイル）
- `validation/` — 検証レポート・調査結果
- `mvp/` — MVP仕様・プロトタイプ設計
- `pitches/` — 事業計画・ピッチ資料

## 実証済みワークフロー知見（まるなげAI案件から）

### 実行フロー
```
1. アイデア創出（ideas/）
   - 3案以上を市場分析付きで並列生成し、推奨案を絞る
   - テンプレートに沿って即記録（粗くてOK）

2. 市場調査・検証（validation/）
   - 統計データは必ず一次ソース（政府白書・業界調査レポート）を引用
   - 「追い風」と「向かい風」の両面を洗い出す
   - 価格戦略は競合の実売価格＋ターゲットの実際の支出傾向データで判断

3. ピッチ資料作成（pitches/）
   - pptxgenjsで自動生成（.jsスクリプト → node実行 → .pptx出力）
   - 必須スライド: 表紙 / 課題 / 解決策 / ビジネスモデル / 市場規模 / 競合分析 / 差別化 / 収益計画 / ロードマップ

4. 判定
   - GO / PIVOT / KILL の3択
   - KILLでも記録は残す（次の事業検討の学びとして）
```

### 事業計画プレゼンのポイント

- **データは必ずソース付き**: 数字には出典（調査名・発行元・年度）を明記。ソースなしの数字は信頼性ゼロ
- **日本市場の主要データ源**:
  - フリーランス人口 → 総務省「就業構造基本調査」、ランサーズ「フリーランス実態調査」
  - AI市場規模 → IDC Japan、総務省「情報通信白書」
  - SaaS市場 → 富士キメラ総研
  - LINE普及 → LINEヤフー公式リリース
  - 中小企業AI導入 → 野村総合研究所
- **価格設定の落とし穴**: フリーランスの62%がSaaS月額3,000円以下。「安い」と思う価格帯と実際の支払い能力にギャップがある
- **pptxgenjs利用**: `/tmp/pptx-build/`にnode_modulesを配置、`node スクリプト.js`で.pptx直接生成

### 他部署との連携
- **dev部門**: MVP実装が必要な場合はdevに依頼。技術選定はdev担当
- **research部門**: 市場調査の深掘りはresearchと連携可能
- **secretary**: 進捗・意思決定は秘書のTODOに報告
